## 2. Правило разбиения больших задач

### 2.1 Почему это важно
```
БОЛЬШАЯ ЗАДАЧА → высокий риск ошибок → сложно отследить → сложно откатить
МАЛЕНЬКИЕ ЗАДАЧИ → низкий риск → легко проверить → легко откатить
```

### 2.2 Алгоритм разбиения
```
1. ПРОЧЕСТЬ задачу полностью
2. ОПРЕДЕЛИТЬ границы компонентов (какие файлы затрагивает)
3. РАЗБИТЬ на подзадачи:
   - Модель → ViewModel → View (MVVM порядок)
   - Сервис → интеграция → тесты
4. ВЫПОЛНЯТЬ по одной подзадаче
5. ПРОВЕРЯТЬ после каждой (компиляция, логика)
6. ТОЛЬКО ПОТОМ переходить к следующей
```

### 2.3 Критерии «большой задачи»
Задача считается **большой** и **требует разбиения**, если:
- ✅ Затрагивает >3 файлов
- ✅ Содержит >5 логических шагов
- ✅ Включает новые компоненты (модель + сервис + ViewModel + View)
- ✅ Изменяет архитектуру (рефакторинг, паттерны)
- ✅ Требует миграции данных

### 2.4 Как оформлять разбиение
```markdown
### Задача: [Название]
**Подзадача 1:** [Описание] → Файлы: [список]
**Подзадача 2:** [Описание] → Файлы: [список]
**Подзадача 3:** [Описание] → Файлы: [список]

Порядок выполнения:
1. Подзадача 1 → проверить компиляцию
2. Подзадача 2 → проверить компиляцию
3. Подзадача 3 → проверить компиляцию
```

### 2.5 Запреты
```
❌ НЕЛЬЗЯ менять >5 файлов за один раз
❌ НЕЛЬЗЯ писать >200 строк кода без промежуточной проверки
❌ НЕЛЬЗЯ делать архитектурные изменения без плана
❌ НЕЛЬЗЯ удалять код без подтверждения
❌ НЕЛЬЗЯ трогать App.xaml (555 строк) без чёткой задачи
```

### 2.6 Обязательные действия после каждой подзадачи
```
1. Проверить синтаксис (скомпилируется ли)
2. Проверить биндинги (если XAML)
3. Проверить зависимости (не сломалось ли другое)
4. Убедиться в Dispose для WebView2
5. Убедиться в async/await корректности
```

## 3. Рабочий процесс AI

### 3.1 Получение задачи
```
1. Прочитать задачу
2. Определить затронутые файлы (использовать WALKTHROUGH.md)
3. Проверить KNOWN_ISSUES.md (нет ли связанных проблем)
4. Проверить SPRINT.md (есть ли уже такая задача)
5. Если задача большая — разбить на подзадачи
6. Уточнить у пользователя если что-то неясно
```

### 3.2 Генерация кода
```
1. ПРОЧЕСТЬ существующий код (read_file)
2. ПОНЯТЬ контекст и стиль
3. НАПИСАТЬ код в том же стиле
4. ПРОВЕРИТЬ синтаксис (скомпилируется ли)
5. ПРОВЕРИЗологические ошибки (null reference, memory leak)
6. ОБЪЯСНИТЬ что было сделано
```

### 3.3 Проверка качества
```
Перед сдачей кода AI проверяет:
□ Синтаксис C# корректен
□ Биндинги XAML соответствуют ViewModel
□ Dispose вызывается где нужно
□ async/await используется правильно
□ Нет дублирования кода (DRY)
□ Обработка ошибок присутствует
□ Debug.WriteLine для логирования
□ INotifyPropertyChanged для binding-свойств
□ RelayCommand для команд
□ Нет утечек памяти (подписка/отписка от событий)
```

## 4. Коммуникация

### 4.1 Стиль общения
- **Кратко и по делу** — без лишних объяснений очевидного
- **Таблицы и чеклисты** — вместо длинных абзацев
- **Код с комментариями** — только к сложным участкам
- **Предупреждения** — если действие рискованное

### 4.2 Формат ответа на задачу
```markdown
**Задача:** [название]
**Затронутые файлы:** [список]

### Что сделано:
1. [Действие] → [файл]
2. [Действие] → [файл]

### Проверить:
- [ ] Компиляция
- [ ] Функциональность

### Замечания:
[Если есть]
```

## 5. Чеклист перед коммитом

```
□ Код компилируется (dotnet build)
□ Нет утечек памяти (WebView2 Dispose)
□ Нет незакрытых task/async void (кроме event handlers)
□ Все binding-свойства имеют INotifyPropertyChanged
□ Все команды через RelayCommand/ICommand
□ XAML не содержит ошибок биндинга
□ Обработка ошибок (try/catch где нужно)
□ Debug.WriteLine для критических операций
□ Нет хардкода путей (использовать AppDataPath)
□ Следует конвенциям из RULES.md
```
