### Принципы:
- **Лучше потратить больше времени и написать стабильный код, чем быстро написать нерабочее говно**

- **Проверяй себя постоянно** — после каждого изменения убеждайся, что оно работает

## 0.2. 📚 Правила работы с кастомными библиотеками

### Что такое «кастомная библиотека»
Любая внешняя NuGet-пакет, SDK, API, или файл который НЕ написан тобой в этом проекте.

### Правило 1: ИЗУЧИ перед использованием
1. **Найди официальную документацию** — через `web_search`
2. **Прочитай примеры использования** — через `web_fetch` страницы документации
3. **Проверь совместимость** — версия пакета, целевой фреймворк, зависимости
4. **НЕ ВЫДУМЫВАЙ** API — вызовы которые не нашёл в документации могут не существовать

### Правило 2: Проверь перед интеграцией
```
1. Как библиотека подключается к проекту? (NuGet, DLL, SDK)
2. Какие namespace/import нужны?
3. Есть ли примеры кода на C#?
4. Совместима ли с .NET 7.0 / WPF / WebView2?
5. Есть ли известные проблемы или ограничения?
```

### Правило 3: Минимальная интеграция
- Подключи только то, что нужно для задачи
- Не тащи всю библиотеку если нужна одна функция
- Оберни вызовы библиотеки в свой сервис/класс — изолируй зависимость

### Правило 4: Обработка ошибок библиотеки
- Все вызовы внешней библиотеки — в try-catch
- Логируй ошибки через `Debug.WriteLine`
- Если библиотека крашится — приложение НЕ должно падать

### Правило 5: Если библиотека не подходит
- Если после изучения документации понял что API не подходит → скажи пользователю
- Предложи альтернативную библиотеку или решение
- НЕ пытайся «подогнать» неправильный API под задачу

### Примеры правильных действий:
```
✅ "Не знаю как работает CoreWebView2.AddWebResourceRequestedFilter → гуглю документацию Microsoft"
✅ "Не уверен что Brushes.Parse существует → проверю через web_search → не существует → использую BrushConverter"
✅ "Нужна библиотека для DoH → найду NuGet пакет → проверю документацию → интегрирую"
```

### Примеры неправильных действий:
```
❌ Придумать метод которого нет (например, webView.InstallExtension())
❌ Использовать API без проверки документации
❌ Игнорировать ошибки совместимости
❌ Подключать библиотеку не проверив как она работает
```

## 3. 💻 Написание кода

### 3.1. Комментарии — обязательно
Каждый **нестрочный** метод/блок кода должен иметь комментарий:

```csharp
// ❌ Плохо — нет контекста
var tab = new TabViewModel(env, searchService, url);
Tabs.Add(tab);
SelectedTab = tab;

// ✅ Хорошо — понятно что, зачем, почему
// Создаём новую вкладку и делаем её активной.
// TabViewModel инициализирует WebView2 асинхронно внутри себя.
var tab = new TabViewModel(env, searchService, url);
Tabs.Add(tab);
SelectedTab = tab; // Установка SelectedTab запускает UpdateFromSelectedTab
```

**Особенно комментируй:**
- Где есть подозрения на нестабильность
- Асинхронный код (async/await, Task)
- Работу с UI-потоком (Dispatcher.Invoke)
- Жизненный цикл объектов (создание/dispose)
- Обход багов/костыли (с пометкой `TODO:` или `HACK:`)

### 3.2. Не выдумывай API
- Не придумывай методы, которых нет в документации
- Если используешь библиотеку — убедись, что метод существует
- Проверь импорты/namespace перед использованием

### 3.3. Стиль кода
- Следуй стилю существующего проекта (именование, отступы, форматирование)
- Не меняй код, который не относится к задаче
- Пиши минимальные изменения — не переписывай всё вокруг

### 3.4. Обработка ошибок
- **НИКОГДА** не оставляй пустые `catch {}`
- Логируй ошибки через `Debug.WriteLine()`
- Показывай пользователю понятные сообщения
- Graceful degradation — лучше частичная работа, чем краш

```csharp
// ❌ Плохо
try { SaveData(); } catch { }

// ✅ Хорошо
try { SaveData(); }
catch (Exception ex)
{
    Debug.WriteLine($"Save failed: {ex.Message}");
    StatusText = "Ошибка сохранения данных";
}
```

### 3.5. Асинхронность
- **Избегай `async void`** — используй `async Task` везде, кроме event handlers
- Для команд создай `AsyncRelayCommand` (если его нет — укажи в TODO)
- Используй `Dispatcher.InvokeAsync` для UI-операций
- Обрабатывай отмену задач

### 3.6. UI и WPF
- Не создавай UI-элементы вне UI-потока
- `Dispatcher.Invoke/InvokeAsync` для любых изменений UI
- WebView2 — тяжёлый объект, не создавай больше необходимого
- При смене `WindowState`/`WindowStyle` — вызывай `InvalidateVisual()` если нужно

### 3.7. Правила WPF Layout — ОБЯЗАТЕЛЬНО
- **Grid ColumnDefinitions**: всегда проверяй, что количество колонок >= максимального `Grid.Column` используемой кнопки
- **StackPanel с Orientation=Horizontal**: элементы не должны выходить за пределы — используй `ScrollViewer` если элементов много
- **Margin/Padding**: всегда указывай явно, не полагайся на дефолты
- **Width/Height**: не используй фиксированные размеры для контейнеров — только `Auto` или `*`
- **Перед изменением UI**: проверь ВСЕ Grid.Row/Column определения и убедись, что элементы не перекрываются
- **После добавления кнопки**: перечитай весь Grid/XAML целиком — убедись, что все элементы на своих местах

## 9. 💡 Паттерны и соглашения

### MVVM
- ViewModel **не должен** знать о конкретных View
- Commands через `ICommand` (RelayCommand)
- INotifyPropertyChanged через `ViewModelBase.Set()`

### Сервисы
- Сервисы — singleton per ViewModel (создаются в MainViewModel)
- Сервисы **не должны** реализовывать INPC (это ответственность VM)
- Хранение данных: `%APPDATA%\GhostBrowser\*.json`

### WPF
- Code-behind только для UI-событий, которые нельзя через команды
- DataTemplate для автоматического отображения View по типу VM
- ContentControl + Binding для переключения контента
