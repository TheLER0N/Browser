## 🎯 Ключевой принцип: БОЛЬШАЯ ЗАДАЧА = МАЛЕНЬКИЕ ШАГИ

**Проблема:** Если дать ИИ огромное задание целиком → он допускает ошибки, теряет контекст, создаёт баги.

**Решение:** Разбить большую задачу на маленькие подпланы. Каждый подплан = одно маленькое изменение. Один подплан = один проход.

**Правило:** 
- ❌ НЕЛЬЗЯ выполнять большую задачу целиком
- ✅ МОЖНО выполнять только маленькие подпланы (1-3 файла, <50 строк)
- ✅ ОБЯЗАТЕЛЬНО записывать каждый подплан в tasks.md перед выполнением
- ✅ ОБЯЗАТЕЛЬНО отмечать выполненные подпланы
- ✅ ОБЯЗАТЕЛЬНО проверять себя после каждого подплана

## 📋 Система подпланов

### 1. Что такое «маленький подплан»

Подплан считается **маленьким**, если:
- Затрагивает **1-3 файла** (не больше)
- Изменяет **<50 строк кода** (если больше — разбить ещё)
- Имеет **конкретный результат** (не «улучшить», а «добавить try-catch в метод X»)
- Можно **проверить** после выполнения (скомпилировать, протестировать)

### 2. Когда разбивать на подпланы

Разбивай на подпланы, если задача:
- Добавляет новую фичу (не багфикс)
- Затрагивает >3 файлов
- Требует изменений в архитектуре
- Содержит слова «рефакторинг», «улучшить», «оптимизировать»
- Содержит >3 пунктов в описании

**НЕ разбивай** на подпланы, если задача:
- Исправление одного бага в одном файле
- Обновление документации
- Простое изменение (1 файл, <10 строк)

### 3. Формат подплана

Каждый подплан записывается в `tasks.md` в формате:

```
### Подплан X.Y: [Название]
- **Родительская задача:** [Название или ссылка]
- **Файлы:** [список файлов]
- **Что сделать:** [конкретное описание, 1-2 предложения]
- **Как проверить:** [команда или действие]
- **Статус:** pending → in_progress → completed
```

## 🔄 Алгоритм работы с большой задачей

### Шаг 1: Понять задачу (ЧТЕНИЕ)

```
1. Прочитать task.md (что хочет пользователь)
2. Прочитать tasks.md (детализация)
3. Прочитать architecture.md (понять контекст)
4. Прочитать design.md / security.md (если касается UI/безопасности)
5. Прочитать rules.md (напомнить правила)
6. Найти все связанные файлы кода (grep/glob)
7. Прочитать каждый связанный файл
```

### Шаг 2: Оценить размер

```
ЕСЛИ задача затрагивает >3 файлов ИЛИ >50 строк:
    → РАЗБИТЬ на подпланы (перейти к Шагу 3)
ИНАЧЕ:
    → Выполнить напрямую (перейти к Шагу 4)
```

### Шаг 3: Разбить на подпланы

```
1. Определить логические части задачи
2. Для каждой части:
   - Какие файлы затронет?
   - Сколько строк изменится?
   - Можно ли проверить отдельно?
3. Если часть ещё большая → разбить ещё
4. Записать ВСЕ подпланы в tasks.md
5. Показать пользователю план (список подпланов)
6. Дождаться подтверждения (если пользователь не против → начать)
```

**Пример разбивки:**

```
Задача: «Добавить валидацию URL»

Подплан 1: Добавить метод валидации в SearchService
  - Файлы: SearchService.cs
  - Что: метод ValidateUrl(string) → bool
  - Проверка: dotnet build

Подплан 2: Вызвать валидацию в TabViewModel.Navigate()
  - Файлы: TabViewModel.cs
  - Что: проверка перед new Uri(), ошибка через Debug.WriteLine
  - Проверка: dotnet build

Подплан 3: Показать ошибку пользователю
  - Файлы: MainViewModel.cs, MainWindow.xaml
  - Что: StatusText = "Некорректный URL"
  - Проверка: dotnet build, ручной тест

Подплан 4: Покрыть тестами
  - Файлы: SearchServiceTests.cs (новый)
  - Что: unit-тесты для ValidateUrl
  - Проверка: dotnet test
```

### Шаг 4: Выполнить подплан

```
1. Найти подплан в tasks.md → статус: pending
2. Изменить статус на: in_progress
3. Прочитать файлы, которые будут изменены (ещё раз!)
4. Внести изменения (минимальные, точные)
5. Добавить комментарии к новому коду
6. Проверить себя (Шаг 5)
7. Изменить статус на: completed
8. Перейти к следующему подплану
```

### Шаг 5: Проверить себя (ПОСЛЕ КАЖДОГО ПОДПЛANA)

```
1. Прочитать изменённый файл целиком
2. Проверить:
   - Все импорты на месте?
   - Нет null reference?
   - Нет утечек ресурсов?
   - Стиль совпадает с проектом?
3. dotnet build (если возможно)
4. Если ошибки → исправить СРАЗУ
5. Если всё ок → пометить подплан completed
```

### Шаг 6: Завершение задачи

```
1. ВСЕ подпланы completed?
2. Обновить tasks.md — отметить задачу выполненной
3. Обновить user-responses.md — записать контекст
4. Обновить AI-файлы (если изменились)
5. Написать отчёт пользователю
```

## 🚨 Критические правила (НЕ НАРУШАТЬ)

### ❌ НЕЛЬЗЯ:
- ❌ Выполнять большую задачу целиком без разбивки
- ❌ Пропускать запись подпланов в tasks.md
- ❌ Менять >3 файлов в одном подплане
- ❌ Менять >50 строк в одном подплане
- ❌ Пропускать проверку после подплана
- ❌ Начинать следующий подплан, не завершив текущий
- ❌ Думать/предполагать — если не уверен, спроси пользователя

### ✅ ОБЯЗАТЕЛЬНО:
- ✅ Прочитать все 10 AI-файлов перед началом
- ✅ Найти все связанные файлы кода (grep/glob)
- ✅ Прочитать каждый связанный файл перед изменением
- ✅ Записать подпланы в tasks.md
- ✅ Помечать статус каждого подплана
- ✅ Проверять себя после каждого подплana
- ✅ Обновить AI-файлы после завершения
- ✅ Написать отчёт пользователю

### Как использовать tasks.md для подпланов

```
## 🆕 Текущая задача: [Название]

### Подплан 1: [Название]
- **Файлы:** [список]
- **Что сделать:** [описание]
- **Как проверить:** [действие]
- **Статус:** ✅ completed

### Подплан 2: [Название]
- **Файлы:** [список]
- **Что сделать:** [описание]
- **Как проверить:** [действие]
- **Статус:** 🔄 in_progress

### Подплан 3: [Название]
- **Файлы:** [список]
- **Что сделать:** [описание]
- **Как проверить:** [действие]
- **Статус:** ⏳ pending
```

### Символы статуса:
- `⏳ pending` — ещё не начат
- `🔄 in_progress` — выполняется сейчас
- `✅ completed` — выполнен и проверен
- `⚠️ blocked` — заблокирован (указать причину)
- `❌ failed` — провален (указать ошибку)

## 🔄 Обратная связь с пользователем

### Когда спрашивать пользователя:
- План разбивки неочевиден (несколько вариантов)
- Подплан заблокирован внешней зависимостью
- Нужно подтверждение архитектурного решения
- Обнаружил непредвиденную проблему

### Когда НЕ спрашивать:
- Баг очевидный (есть один правильный фикс)
- Задача чётко определена
- Можно вывести из контекста кода
